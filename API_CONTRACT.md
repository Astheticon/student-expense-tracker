# Student Expense Tracker – Extended API Specification

This document defines ALL required data structures, validation rules, logic formulas,
error handling, and behavior expectations.

Both Web and Mobile implementations MUST strictly follow this specification.

This document is the single source of truth.

---

# 1️⃣ System Overview

Application Type:
Personal Student Expense Tracker

Core Features:
- User Authentication
- Add Income
- Add Expense
- View Transactions
- View Financial Summary

Currency:
INR (₹)

All currency values:
- Stored as numbers (no symbols)
- Displayed with 2 decimal precision in frontend/mobile
- Backend does NOT store currency symbols

---

# 2️⃣ Authentication Rules

Authentication Method:
JWT-based OR session-based (implementation-specific)

Required Behavior:
- All transaction endpoints must require authentication
- Unauthorized requests must return HTTP 401

Standard Unauthorized Response:

Status: 401
{
  "error": "Unauthorized"
}

---

# 3️⃣ Transaction Data Model (STRICT)

Transaction Object:

{
  "id": "string",
  "type": "income | expense",
  "category": "string",
  "amount": number,
  "note": "string | null",
  "date": "ISO string",
  "createdAt": "ISO string"
}

Field Definitions:

id:
- Unique identifier
- UUID format recommended

type:
- Must be EXACTLY "income" or "expense"
- Case-sensitive
- Lowercase only

category:
- Non-empty string
- Max length: 50 characters
- Case-sensitive behavior must be consistent across platforms

amount:
- Must be positive number (> 0)
- Max value: 1,000,000 (safety limit)
- Up to 2 decimal places allowed
- Negative numbers are NOT allowed

note:
- Optional
- Max length: 255 characters
- Can be null

date:
- ISO 8601 format
- Example: 2026-02-23T00:00:00.000Z
- Must be stored in UTC
- Backend does NOT perform timezone conversion

createdAt:
- Automatically generated by backend
- UTC timestamp

---

# 4️⃣ Validation Rules (MANDATORY)

For POST /transactions:

Validation Conditions:
- type must exist
- category must exist
- amount must exist
- date must exist
- amount > 0
- type must be "income" or "expense"

If validation fails:

Status: 400
{
  "error": "Invalid input"
}

No partial acceptance allowed.

---

# 5️⃣ Add Transaction

POST /api/transactions

Request Body:

{
  "type": "income",
  "category": "Food",
  "amount": 200,
  "note": "Lunch",
  "date": "2026-02-23T00:00:00.000Z"
}

Response (Success):

Status: 201
{
  "message": "Transaction created"
}

Response (Failure):
Status: 400
{
  "error": "Invalid input"
}

---

# 6️⃣ Get Transactions

GET /api/transactions

Behavior Rules:
- Only return transactions for authenticated user
- Order by createdAt DESC (newest first)
- No pagination required for MVP
- Return empty array if no transactions

Response:

Status: 200
[
  {
    "id": "uuid",
    "type": "income",
    "category": "Food",
    "amount": 200,
    "note": "Lunch",
    "date": "2026-02-23T00:00:00.000Z",
    "createdAt": "2026-02-23T10:00:00.000Z"
  }
]

---

# 7️⃣ Summary Endpoint

GET /api/summary

Calculation Rules:

totalIncome  = sum of all transactions where type = "income"
totalExpense = sum of all transactions where type = "expense"
balance      = totalIncome - totalExpense

Balance MUST be calculated exactly as:

balance = totalIncome - totalExpense

No rounding during calculation.
Rounding only applied in UI display.

Response:

Status: 200
{
  "totalIncome": number,
  "totalExpense": number,
  "balance": number
}

If no transactions:

{
  "totalIncome": 0,
  "totalExpense": 0,
  "balance": 0
}

---

# 8️⃣ Error Handling Standard

All error responses MUST follow this structure:

{
  "error": "Error message here"
}

No HTML responses.
No plain text responses.
Always JSON.

Standard Status Codes:

400 → Invalid input
401 → Unauthorized
404 → Not found
500 → Server error

---

# 9️⃣ Date & Time Rules

- All stored dates must be UTC
- Backend does NOT convert timezone
- Frontend/mobile responsible for local formatting
- ISO string required everywhere

Example:
2026-02-23T00:00:00.000Z

---

# 🔟 Sorting Rules

Transactions:
- Sorted by createdAt DESC

No client-side sorting assumptions allowed.

---

# 11️⃣ Edge Case Behavior

If:
- No transactions → return empty array
- Only income exists → totalExpense = 0
- Only expense exists → totalIncome = 0
- Balance can be negative (allowed)

Negative balance is VALID.

---

# 12️⃣ Consistency Rules

Both implementations must ensure:

- Same calculation formula
- Same validation rules
- Same status codes
- Same field names
- Same case sensitivity

Changing any field requires updating this document first.
